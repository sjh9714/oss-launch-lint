좋아요. 지금 상태는 **“레포 MVP/품질 패키징 완료”**이고, 아직 **“실제 공개 런칭/홍보*는 시작 전**입니다. 이제부터는 Codex에게 한 번 더 시키거나, 직접 터미널에서 아래 순서대로 밀면 됩니다.

핵심 주의점 하나: **README에 `npm install -g oss-launch-lint` 같은 설치 문구가 있는데 npm publish를 안 했다면, 그대로 공개하면 첫 사용자가 실패합니다.** 공개 전에 `README` 설치 섹션이 “GitHub에서 클론해서 실행”인지, 아니면 실제 npm 배포까지 끝났는지 반드시 맞춰야 합니다.

`YOUR_USERNAME`이 나오면 전부 실제 GitHub 아이디로 바꾸세요.

```bash
# 예: GitHub 아이디가 sungjh라면
perl -pi -e 's/YOUR_USERNAME/sungjh/g; s/YOUR_GITHUB_USERNAME/sungjh/g' \
```

그다음 변경사항 커밋:

이미 커밋되어 있으면 `nothing to commit`이 나올 수 있고, 그건 괜찮습니다.

## 2. `main` 브랜치로 정리

지금 브랜치가 `codex/oss-launch-lint-mvp`라면, 공개 레포 기본 브랜치는 `main`으로 두는 게 좋습니다.

```bash
# main이 없으면 현재 상태에서 main 생성
if git show-ref --verify --quiet refs/heads/main; then
  git switch main
  git merge --no-ff codex/oss-launch-lint-mvp
else
  git switch -c main
fi
```

다시 한 번:

```bash
npm test && npm run lint && npm run build
```

## 3. GitHub 공개 레포 생성 + push

GitHub CLI를 쓰면 기존 로컬 디렉터리에서 바로 public repo를 만들고 push할 수 있습니다. 공식 `gh repo create` 문서 기준으로 `--source`는 로컬 저장소를 원격 repo 생성에 쓰고, `--push`는 로컬 커밋을 새 repo에 push합니다. ([GitHub CLI][1])

```bash
gh auth status || gh auth login

gh repo create oss-launch-lint \
  --public \
  --source=. \
  --remote=origin \
  --push \
  --description "Audit whether a GitHub repo is ready for an ethical open-source launch."
```

조직 계정에 만들 거면:

```bash
gh repo create YOUR_ORG/oss-launch-lint \
  --public \
  --source=. \
  --remote=origin \
  --push \
  --description "Audit whether a GitHub repo is ready for an ethical open-source launch."
```

## 4. 설명, 토픽, 설정 넣기

`gh repo edit`는 repo 설정 수정용이고, 현재 매뉴얼에 `--description`, `--homepage`, `--add-topic` 옵션이 있습니다. ([GitHub CLI][2])

```bash
OWNER="$(gh api user --jq .login)"
REPO="$OWNER/oss-launch-lint"

gh repo edit "$REPO" \
  --description "Audit whether a GitHub repo is ready for an ethical open-source launch." \
  --homepage "https://github.com/$REPO" \
  --enable-issues=true \
  --enable-wiki=false \
  --add-topic open-source \
  --add-topic cli \
  --add-topic github \
  --add-topic developer-tools \
  --add-topic maintainer-tools \
  --add-topic typescript \
  --add-topic oss \
  --add-topic launch \
  --add-topic checklist
```

이후 GitHub 웹에서 확인할 것:

```text
https://github.com/YOUR_USERNAME/oss-launch-lint
```

확인 포인트는 README 첫 화면입니다.

| 체크         | 기준                                       |
| ---------- | ---------------------------------------- |
| 첫 5줄       | 무엇인지, 누구에게 유용한지 바로 보이는가                  |
| 설치         | 실제로 동작하는 설치 방법인가                         |
| Quickstart | 복붙해서 실행 가능한가                             |
| 예시 출력      | Markdown/JSON 결과가 보이는가                   |
| CTA        | “피드백 주세요”가 자연스러운가                        |
| 배지         | CI, license, npm 또는 package 상태가 거짓말이 아닌가 |

## 5. v0.1.0 릴리즈 만들기

GitHub CLI의 `gh release create`는 새 GitHub Release를 만드는 명령이고, 태그가 없으면 기본 브랜치 최신 상태 기준으로 태그를 만들 수 있지만, 안정적으로 하려면 로컬에서 annotated tag를 만들고 push한 뒤 release를 만드는 게 좋습니다. ([GitHub CLI][3])

```bash
git tag -a v0.1.0 -m "v0.1.0"
git push origin v0.1.0

gh release create v0.1.0 \
  --title "oss-launch-lint v0.1.0" \
  --notes-file docs/release-notes-v0.1.0.md
```

README가 npm 설치를 전제로 한다면 여기서 둘 중 하나를 선택해야 합니다.

**A안: npm publish 안 함**

README 설치 섹션을 이렇게 바꾸세요.

````md
## Quickstart

```bash
git clone https://github.com/YOUR_USERNAME/oss-launch-lint.git
cd oss-launch-lint
npm install
npm run build
node dist/cli.js .
````

````

**B안: npm publish 함**

`package.json`의 `name`, `bin`, `files`, `version`, `license`를 확인한 뒤:

```bash
npm login
npm publish --dry-run
npm publish
````

개인적으로는 처음에는 **A안으로 공개 → 반응 있으면 npm publish**가 더 안전합니다. 설치 실패 리스크가 줄고, v0.1.0에서 피드백을 빨리 받을 수 있습니다.

## 7. 홍보 문구는 이렇게 가는 게 좋습니다

GitHub 정책상 자동화된 과도한 bulk activity, fake accounts, automated inauthentic activity, automated starring/following 같은 rank abuse는 금지되어 있습니다. 그래서 “스타 눌러주세요”보다 “피드백 주세요”가 안전하고 반응도 낫습니다. ([GitHub Docs][4])

### X / Threads / LinkedIn용

```text
I built oss-launch-lint: a tiny CLI that checks whether a GitHub repo is ready for an open-source launch.

It audits README, license, CI, community files, release prep, and ethical promotion copy, then generates a Markdown/JSON report.

I made it because most small OSS projects fail at the boring launch details, not the code.

Repo:
https://github.com/YOUR_USERNAME/oss-launch-lint

Would love feedback from maintainers: what should it check next?
```

### Hacker News / Show HN용

```text
Show HN: oss-launch-lint – Check if your GitHub repo is ready for an OSS launch

I built a small TypeScript CLI that audits a repo for open-source launch readiness.

It checks things like README gaps, license, CI, community files, release prep, and ethical promotion copy, then outputs a Markdown or JSON report.

I made it for small tools and AI-built side projects where the code exists, but the repo is not actually ready to share.

Repo:
https://github.com/YOUR_USERNAME/oss-launch-lint

Feedback welcome, especially from maintainers.
```

### Reddit/dev.to용 짧은 버전

```text
I built a small CLI for maintainers: oss-launch-lint.

It checks whether a GitHub repo is ready to launch as an open-source project: README, license, CI, community files, release notes, and promotion copy.

The goal is to catch the boring-but-important launch gaps before you post the repo publicly.

Repo:
https://github.com/YOUR_USERNAME/oss-launch-lint

What checks would you add?
```
