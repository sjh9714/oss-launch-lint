# Repository quality checklist

## Product clarity
- [ ] Repo name is memorable and describes the tool.
- [ ] Description is under 120 characters and explains the user benefit.
- [ ] README first screen shows what it does, who it is for, and a concrete example.
- [ ] A new user can run the first useful command within 2 minutes.
- [ ] Demo output is visible in README.

## Code quality
- [ ] Core behavior is tested.
- [ ] Lint/build/test commands are documented.
- [ ] CI runs on pull requests and main branch.
- [ ] No secrets or local-only paths are committed.
- [ ] Error messages are actionable.

## Open-source hygiene
- [ ] LICENSE exists.
- [ ] CONTRIBUTING.md exists.
- [ ] CODE_OF_CONDUCT.md exists.
- [ ] SECURITY.md exists.
- [ ] CHANGELOG.md exists.
- [ ] Issue templates exist.
- [ ] Roadmap exists.

## README sections
- [ ] What the project does
- [ ] Why it is useful
- [ ] Installation
- [ ] Quickstart
- [ ] Examples
- [ ] Output screenshot/GIF or text demo
- [ ] Configuration/options
- [ ] Roadmap
- [ ] Contributing
- [ ] Support/help
- [ ] License

## Launch readiness
- [ ] Repo topics chosen.
- [ ] v0.1.0 release notes drafted.
- [ ] Promotion copy drafted for each approved channel.
- [ ] Metrics tracker created.
- [ ] Manual posting checklist prepared.
- [ ] Follow-up improvement backlog prepared.

## Ethical promotion guardrails
- [ ] No fake stars.
- [ ] No automated stars/follows.
- [ ] No star exchanges.
- [ ] No giveaways or incentives for stars.
- [ ] No mass DMs.
- [ ] No irrelevant GitHub issue/PR comments.
- [ ] Community rules checked before posting.
