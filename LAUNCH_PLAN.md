# Ethical 7-day launch plan

## Principle
The goal is not to “get stars” through pressure or manipulation. The goal is to ship something genuinely useful, put it in front of the right users, ask for feedback, and make it easy for people to star only if they find it useful.

## Day 0 — Build the shareable MVP
- Implement the smallest working feature set.
- Add tests and CI.
- Write a README that shows the result immediately.
- Add an example report or demo fixture.
- Prepare screenshots, GIF instructions, or terminal recording instructions.

## Day 1 — Polish the repository page
- Add repo description.
- Add relevant GitHub topics.
- Add badges only if they are real and useful.
- Add community files.
- Add issue templates.
- Add roadmap with 3-5 concrete next features.
- Create v0.1.0 release notes.

## Day 2 — Soft launch to warm contacts
- Share privately with 3-5 developer friends or communities where you already participate.
- Ask for bugs, confusing parts, and missing use cases.
- Fix the top issues before wider posting.
- Do not ask for stars as the main request.

## Day 3 — Public launch wave 1
- Post to your own GitHub profile / pinned repo.
- Post one concise thread/post on your main social account.
- Publish a short article or post explaining the problem, before/after example, and what you learned.
- Record URLs and initial metrics.

## Day 4 — Community launch wave 2
- Post only in communities where project sharing is allowed and genuinely relevant.
- Tailor the message to the community.
- Ask for feedback and use cases.
- Reply to comments quickly and respectfully.

## Day 5 — Iterate based on feedback
- Fix issues raised by users.
- Add one high-signal improvement.
- Update README with clearer examples.
- Thank contributors/commenters.

## Day 6 — Follow-up post
- Share a useful update, not a duplicate launch post.
- Example: “I added JSON output and a GitHub Action example after feedback.”
- Include the repo link only where appropriate.

## Day 7 — Measure and decide next step
- Record star count, unique visitors, clones, referrers, issues, and comments.
- Identify the highest-converting channel.
- Decide whether to continue, pivot, or write a technical deep dive.

## 16-star path hypothesis
A reasonable path to 16 legitimate stars is:
- 5-8 from warm developer contacts who actually inspect the repo
- 3-5 from one relevant community post
- 3-5 from social/article visibility
- 1-3 from GitHub discovery, profile pinning, or follow-up improvements

This is not guaranteed. Treat 16 stars as an outcome metric, not a promise.
